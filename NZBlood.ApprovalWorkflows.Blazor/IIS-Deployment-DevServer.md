# IIS Deployment - Dev Server

This document captures the expected IIS deployment configuration for the Blazor approval workflow on the `NZBLOOD` dev server.

## Target

- Server: `NZBLOOD`
- Hosting: IIS hosting ASP.NET Core / Blazor Server
- Protocol: HTTP only at this stage
- Framework: .NET 8
- App mode: SQL-backed, not mock data
- SQL authentication: SQL user/password connection string

## Prerequisites

Install on `NZBLOOD` before deployment:

- .NET 8 Hosting Bundle, including the ASP.NET Core Runtime.
- IIS with ASP.NET Core Module support from the Hosting Bundle.
- Network access from `NZBLOOD` to the GP/NZBS SQL Server.
- SQL login with permissions to read the workflow views/tables and execute the workflow stored procedures.

After installing the Hosting Bundle, restart IIS:

```powershell
iisreset
```

## Recommended File Locations

Suggested deployment folder:

```text
C:\inetpub\wwwroot\NZBlood.ApprovalWorkflows.Blazor
```

Suggested IIS logs:

```text
C:\inetpub\logs\LogFiles
```

Suggested app-specific diagnostic/log folder if file logging is later added:

```text
C:\inetpub\logs\NZBlood.ApprovalWorkflows.Blazor
```

Current expectation: persistent application logging should be written to SQL Server, not local files.

## Publish Files

Publish from the project folder:

```powershell
dotnet publish .\NZBlood.ApprovalWorkflows.Blazor.csproj -c Release -o .\publish
```

Copy the contents of:

```text
NZBlood.ApprovalWorkflows.Blazor\publish\
```

to:

```text
C:\inetpub\wwwroot\NZBlood.ApprovalWorkflows.Blazor\
```

The deployed folder should include:

- `NZBlood.ApprovalWorkflows.Blazor.dll`
- `NZBlood.ApprovalWorkflows.Blazor.exe`
- `NZBlood.ApprovalWorkflows.Blazor.deps.json`
- `NZBlood.ApprovalWorkflows.Blazor.runtimeconfig.json`
- `web.config` generated by publish
- `appsettings.json`
- `wwwroot\`
- required dependency DLLs

Do not deploy development-only output folders such as:

- `bin\`
- `obj\`
- `.verify-build\`
- `Scripts\`
- `Sql Objects\`
- `Sql Sample Data\`

## IIS Site Or Application

Either create a dedicated IIS site or an application under an existing internal site.

Recommended dedicated site:

- Site name: `NZBlood Approval Workflows`
- Physical path: `C:\inetpub\wwwroot\NZBlood.ApprovalWorkflows.Blazor`
- Binding type: `http`
- Host name: confirm with infrastructure team, for example `approvals-dev.nzblood.local`
- Port: `80` unless another dev binding is required

If deploying as an application under an existing site:

- Application alias: `ApprovalWorkflows`
- Physical path: `C:\inetpub\wwwroot\NZBlood.ApprovalWorkflows.Blazor`
- URL example: `http://NZBLOOD/ApprovalWorkflows`

## Application Pool

Recommended app pool:

- Name: `NZBlood.ApprovalWorkflows.Blazor`
- .NET CLR version: `No Managed Code`
- Managed pipeline mode: `Integrated`
- Start mode: `AlwaysRunning`
- Enable 32-bit applications: `False`
- Identity: start with `ApplicationPoolIdentity` unless infrastructure requires a domain service account
- Idle timeout: consider `0` for always-on internal workflow use, or leave the IIS default if preferred
- Recycling: schedule outside business hours if a fixed recycle is required

If a domain service account is used for the app pool identity:

- Grant it read/execute access to the deployment folder.
- Confirm it can read any machine-level configuration used for connection strings or secrets.
- SQL access is still expected to use the configured SQL login, not Windows trusted delegation.

## Authentication

The app has no separate application login. It uses the Windows/AD identity from the request and compares the short account name to `nzbWFPPVApproverList.AccountName`.

Recommended IIS authentication:

- Windows Authentication: `Enabled`
- Anonymous Authentication: `Disabled`

Confirm the request identity reaches ASP.NET Core as something like:

```text
NZBLOOD\HyndmanR
```

The app normalises this to the short account name, for example:

```text
HyndmanR
```

That value is compared to SQL approver/admin/manager records.

## App Settings

For `NZBLOOD` deployment, `appsettings.json` should contain:

```json
{
  "ApprovalWorkflow": {
    "DomainName": "NZBLOOD",
    "AppName": "Approvals Workflow",
    "UseMockData": false,
    "TargetDevServer": "NZBLOOD",
    "RequireHttpsRedirection": false,
    "UserIdFormat": "SamAccountName"
  },
  "ConnectionStrings": {
    "ApprovalWorkflow": "Server=<sql-server>;Database=NZBS;User ID=<sql-user>;Password=<sql-password>;TrustServerCertificate=True;"
  }
}
```

Notes:

- `UseMockData` must be `false` on the dev server.
- `RequireHttpsRedirection` should remain `false` while the dev server is HTTP-only.
- The deployed connection string should use a SQL login.
- Do not commit live credentials to source control.
- `appsettings.Development.json` is for local development and should not be used to override the IIS deployment into mock mode.

## Syncfusion

`Program.cs` already registers the Syncfusion license when `Syncfusion:LicenseKey` is present.

Configure the license through one of:

- `appsettings.json` on the dev server
- IIS environment variable
- another secure deployment configuration mechanism

Avoid committing the live license key if it is treated as a secret.

## File And Folder Permissions

Grant the app pool identity:

- Read and execute on `C:\inetpub\wwwroot\NZBlood.ApprovalWorkflows.Blazor`
- Read on all deployed files

Write access is not currently required for normal attachment handling because attachments are read from SQL and streamed to the browser.

Browser downloads are controlled by each user's browser settings, not by a server-side download folder.

## SQL Permissions

The configured SQL login needs:

- Select access to:
  - `nzbWFPPVDashboard`
  - `nzbWFPPVApproverList`
  - `nzbWFPPVTransactionLne`
  - `nzbWFPPVTransactionHdr`
  - `nzbWFPPVInvoiceHistory`
  - `nzbWFPPVInvoiceStatus`
  - `nzbPPVSetup`
  - `nzbApprovalAdmins`
  - `CO00101`
  - `CO00102`
  - `coAttachmentItems`
- Execute access to:
  - `nzbWFPPVReceiptApproval`
  - `nzbWFPPVReceiptReject`
  - `nzbWFPPVReceiptReset`
  - `nzbWFPPVCreateReceiptManagerApproval`

The procedures update GP/workflow tables and call GP posting-related objects, so their execution context and required downstream permissions must be validated with SQL/GP administration.

## Smoke Test

After deployment:

1. Browse to the app over HTTP.
2. Confirm the header user shows the current AD user.
3. Confirm the dashboard shows only receipts where the user's short AD account exists in `nzbWFPPVApproverList.AccountName`.
4. Confirm users with no matching approvals see the empty approvals message.
5. Open a receipt and verify:
   - Header/detail data loads.
   - PO lines load.
   - Attachment links download from SQL binary content.
   - PO PDF links download when `POBinary` is present.
6. Validate approve/reject/reset in a safe dev scenario.
7. Confirm manager approval flow creates the manager approver record when required.

## Rollback

To roll back the web deployment:

1. Stop the IIS site or app pool.
2. Restore the previous published folder backup.
3. Start the app pool/site.

Backend SQL objects should not be modified in place for this UI rollout. If SQL-side changes become necessary, create new objects or wrapper procedures and point the app at those after validation.
